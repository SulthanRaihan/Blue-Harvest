BAB 1 — Pendahuluan

1.1 Latar Belakang

Sektor akuakultur, khususnya budidaya tambak ikan bandeng, nila, dan udang vaname, merupakan salah satu sumber pendapatan utama bagi petambak di Indonesia. Namun, pengelolaan tambak saat ini masih banyak yang bersifat manual dan bergantung pada pengalaman individu tanpa dukungan data yang sistematis. Hal ini menyebabkan ketidakmampuan dalam mendeteksi risiko lebih awal, keterlambatan respons terhadap perubahan kualitas air, serta minimnya rekam jejak operasional yang dapat digunakan untuk evaluasi dan pengambilan keputusan.

Sistem informasi manajemen tambak berbasis digital diperlukan untuk menjawab tantangan tersebut. Dengan memanfaatkan teknologi Internet of Things (IoT), pemrosesan data berbasis aturan (rule-based), dan antarmuka web yang dapat diakses dari berbagai perangkat, petambak dapat memantau kondisi kolam secara real-time, mendapatkan rekomendasi berbasis risiko, serta mengelola seluruh siklus budidaya dari perencanaan hingga distribusi dalam satu platform terintegrasi.

1.2 Tujuan Dokumen

Dokumen Perancangan Sistem (System Design Document) ini bertujuan untuk:

  • Mendeskripsikan arsitektur dan perancangan teknis sistem Blue Harvest secara menyeluruh.
  • Menyajikan model proses bisnis, aliran data, struktur basis data, dan arsitektur teknologi yang akan diimplementasikan.
  • Menjadi acuan bagi tim pengembang selama fase implementasi dan pengujian sistem.
  • Menjadi dokumentasi teknis pendukung laporan tugas akhir (skripsi).

1.3 Ruang Lingkup Sistem

Sistem Blue Harvest mencakup fungsionalitas berikut:

  • Perencanaan Budidaya: Input rencana tebar (komoditas, luas kolam, modal, jumlah benih) dan simulasi skoring risiko berbasis aturan.
  • Operasional Harian: Pencatatan pemberian pakan, pengelolaan hama/penyakit, sampling pertumbuhan, dan kualitas air manual.
  • Monitoring IoT: Pembacaan data sensor real-time (pH, DO, suhu, salinitas) dengan notifikasi alarm otomatis bila parameter menyimpang.
  • Panen & Pasca Panen: Pencatatan hasil panen (bobot, grade, pendapatan) dan quality control.
  • Distribusi & Pemasaran: Manajemen distribusi produk ke pasar/pengepul/restoran dan portal pemesanan bagi pelanggan.
  • Laporan & Evaluasi: Generate laporan otomatis (keuangan, produksi, kinerja) yang dapat diekspor ke PDF.
  • Manajemen Pengguna: Autentikasi berbasis role (Petambak, Admin, Owner, Pelanggan).

1.4 Aktor Sistem

No.AktorPeran
1 Petambak / Operator Pelaku utama di lapangan. Menginput data operasional harian, memantau dashboard, dan menerima notifikasi alarm.
2 Admin Mengelola data master (kolam, pengguna, konfigurasi sensor), memverifikasi data, dan mengakses laporan sistem.
3 Manajemen / Owner Menyetujui rencana budidaya, melihat laporan keuangan dan analitik kinerja, serta mengevaluasi profitabilitas.

5 Sensor IoT Perangkat fisik (ESP32 + sensor pH/DO/Suhu/Salinitas) yang mengirimkan data kualitas air ke sistem secara otomatis.

BAB 2 — Gambaran Umum Sistem

2.1 Deskripsi Sistem

Blue Harvest adalah sistem informasi manajemen tambak berbasis web yang dirancang untuk membantu petambak, admin, dan manajemen dalam mengelola siklus budidaya akuakultur secara terintegrasi. Sistem ini dibangun di atas platform Supabase dengan frontend Next.js (PWA), sehingga dapat diakses dari perangkat apapun — komputer, tablet, maupun smartphone — tanpa memerlukan aplikasi terpisah.

Sistem mendukung tiga komoditas utama: ikan bandeng, ikan nila, dan udang vaname, masing-masing dengan parameter kualitas air dan standar FCR (Feed Conversion Ratio) yang berbeda. Data dari sensor IoT (opsional) dapat dikirimkan langsung ke sistem untuk monitoring real-time, sementara data operasional dapat diinput secara manual oleh operator di lapangan.

2.2 Fitur Utama

No.ModulDeskripsi
1 Perencanaan & Skoring Risiko Input rencana tebar, simulasi skoring risiko rule-based (Best/Middle/Worst Case), approval oleh Owner.
2 Operasional Harian Log harian pemberian pakan, catatan hama/penyakit, tindakan, dan kualitas air manual.
3 Monitoring IoT Dashboard real-time pH, DO, suhu, salinitas dari sensor ESP32. Alarm otomatis saat parameter di luar ambang.
4 Sampling Pertumbuhan Pencatatan sampling mingguan (rata-rata bobot, estimasi populasi, FCR aktual).
5 Panen & Pasca Panen Pencatatan hasil panen (bobot total, grade A/B/C, harga per kg, pendapatan), QC, dan pengecekan kualitas.
6 Distribusi & Pemasaran Pencatatan distribusi ke pelanggan (pengepul, pasar, restoran, eksportir), portal pemesanan publik.
7 Laporan & Evaluasi Generate laporan otomatis periodik (keuangan, produksi, kinerja), ekspor PDF, analitik grafik.
8 Manajemen Pengguna Registrasi, login, manajemen role (Petambak/Admin/Owner/Pelanggan), konfigurasi perangkat sensor.

2.3 Modul Skoring Risiko (Rule-Based)

Modul skoring risiko adalah fitur unggulan Blue Harvest. Sistem mengevaluasi empat kategori faktor risiko budidaya menggunakan metode matriks risiko sederhana (Potensi × Dampak).

Faktor Risiko yang Dinilai

  • Risiko Hama & Penyakit — kemungkinan serangan organisme pengganggu
  • Risiko Cuaca & Iklim — potensi dampak cuaca ekstrem terhadap kolam
  • Risiko Pasar & Harga — volatilitas harga komoditas di pasar
  • Risiko Operasional / SDM — kapasitas tenaga kerja dan ketersediaan sarana produksi

Formula Skoring

Skor per Faktor = Nilai Potensi (1–5) × Nilai Dampak (1–5)
Total Skor = Σ Skor seluruh faktor

Klasifikasi Hasil

Total SkorKategoriKeterangan
≤ 10 BEST CASE Risiko rendah — budidaya dapat dilanjutkan dengan pengawasan standar.
11 – 20 MIDDLE CASE Risiko sedang — diperlukan langkah mitigasi spesifik sebelum melanjutkan.
> 20 WORST CASE Risiko tinggi — disarankan untuk meninjau ulang rencana atau menunda budidaya.

Alur proses pada Blok B adalah sebagai berikut:

7.3 Komponen Arsitektur

A. Zona IoT — Perangkat Lapangan

Perangkat lapangan terdiri dari sensor kualitas air (pH, DO, Suhu, Salinitas) yang terhubung ke mikrokontroler ESP32/ESP8266. Mikrokontroler membaca data sensor secara berkala dan mengirimkannya ke Supabase melalui koneksi WiFi.

B. Supabase Platform — Backend Terpadu

KomponenFungsi
PostgREST REST API otomatis dari skema PostgreSQL. Digunakan oleh Next.js dan ESP32 untuk operasi CRUD data.
Realtime Engine Streaming perubahan data via WebSocket. Next.js subscribe ke channel kolam untuk live update dashboard.
Auth Service Manajemen token JWT, sesi pengguna, dan Row Level Security (RLS) untuk keamanan data per role.
PostgreSQL Database Penyimpanan utama seluruh data sistem (14 tabel sesuai ERD).
File Storage Bucket S3-compatible untuk laporan PDF dan foto dokumentasi.
fn_scoring_risiko() Edge Function — menghitung skor risiko per faktor (Potensi × Dampak) dan mengklasifikasikan hasilnya.
fn_kirim_notifikasi() Edge Function — trigger otomatis saat data sensor melebihi threshold, mengirimkan notifikasi ke tabel NOTIFIKASI.
fn_generate_laporan() Edge Function — mengagregasi data panen, operasional, dan distribusi, lalu menghasilkan laporan dan mengekspornya ke PDF.

C. Zona Klien — Aplikasi Web

Frontend dibangun dengan Next.js dan dikonfigurasi sebagai Progressive Web App (PWA) sehingga dapat diinstall dan diakses offline. Terdapat dua entry point:

  • Next.js Dashboard Utama — aplikasi internal untuk Petambak, Admin, dan Owner dengan autentikasi penuh. Terhubung ke Supabase via Supabase SDK (JavaScript) untuk REST, Realtime, Auth, dan Storage.
  • Portal Pelanggan — halaman publik Next.js untuk Pelanggan/Pembeli yang hanya mengakses data produk dan distribusi tanpa autentikasi penuh.

7.4 Jalur Integrasi IoT

Sistem mendukung dua jalur pengiriman data IoT, disesuaikan dengan kebutuhan dan kemampuan client:

Jalur A — Sederhana (HTTP Batch POST)
ESP32 mengirimkan data sensor langsung ke Supabase REST API via HTTP POST ke endpoint /rest/v1/kualitas_air. Cocok untuk implementasi awal atau jika client tidak memerlukan streaming real-time dengan latensi rendah. Tidak membutuhkan infrastruktur tambahan.
Jalur B — Real-time Streaming (MQTT)
ESP32 mem-publish data ke MQTT Broker (Mosquitto / HiveMQ). Node.js Bridge Service subscribe ke broker dan mem-push data ke Supabase REST API. Jalur ini bersifat opsional dan diaktifkan jika client menginginkan monitoring dengan frekuensi tinggi dan latensi rendah. Ditandai dengan garis putus-putus pada diagram arsitektur.
